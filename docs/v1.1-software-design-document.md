## 1. Product Requirements

Ship the Guest ATS Resume Checker as the platform's public hero feature, then reposition the homepage/navigation around it, then invest in the broader premium-SaaS brand experience — as three independently deployed, independently verified phases.

| Phase | Deliverable | Status |
|---|---|---|
| **1A** | Guest ATS engine — ephemeral, no-signup ATS checker: upload → score → full results → CTA | **Approved, starting now** |
| **1B** | Homepage & navigation repositioning around the live Guest ATS experience | Approved, sequenced after 1A is live-verified |
| **1C** | Premium SaaS branding, visual identity, marketing pages, testimonials, conversion optimization | Approved, sequenced after 1B is live-verified |

Each phase completes the full continuous-deployment cycle (lint → build → tests → commit → push → deploy → live verification → report commit hash) **before the next phase's implementation begins.** Phase 1B does not start until 1A is verified live; 1C does not start until 1B is verified live.

**Cross-cutting principle for all three phases** (and for the rest of Version 1.1 beyond this document): every architectural decision should move the project toward being a world-class AI Career & Talent Platform, not just an ATS with a free-trial gimmick bolted on. Concretely, this means:
- Prefer reusable components, reusable AI-service patterns, reusable layouts, and reusable public-page scaffolding over single-purpose, throwaway implementations — *without* pre-building infrastructure for tools that don't exist yet (see §6 — there's a real line between "don't paint into a corner" and "don't build a runtime for nine unspecified future tools").
- Avoid any decision that would require re-architecting when tool #2 (whichever it turns out to be) ships.
- Where a choice is genuinely close, prefer the option that keeps the platform's story coherent as "AI Career Platform" rather than "ATS with extras."

## 2. Technical Architecture

### 2.1 Baseline (unchanged, re-confirmed)

- Next.js 15.4.11, App Router, React 19, TypeScript strict, deployed via `next build && next start` on Vercel.
- `src/middleware.ts` → `src/lib/supabase/middleware.ts`: single middleware, session refresh + role-based route protection for `/candidate`, `/recruiter`, `/admin` only. Everything else is already implicitly public — no middleware change needed for Phase 1A, same as `/jobs` today.
- RLS is the authorization boundary for every table; `resumes`/`ats_scores`/`candidates` all require `auth.uid()` matching the owning row. No anonymous-insert policy exists anywhere in that chain, and Phase 1A does not add one — it doesn't touch those tables at all.
- AI infra (`src/lib/ai/*`): six hardened functions (embeddings, resume-parser, ats-scorer, job-matcher, screening, resume-improver), all with graceful OpenAI-failure fallback (fixed and live-verified this engagement). Reused unmodified for the guest flow.
- Rate limiting (`src/lib/rate-limit.ts`): in-memory, not distributed-safe across Vercel's serverless instances — used only as a cheap first-line burst check; the authoritative guest gate is Postgres-backed (§7).
- Migrations `0001`–`0014` applied; next is `0015`.

### 2.2 New architecture components (Phase 1A)

1. **Guest identity layer** — httpOnly, Secure, SameSite=Lax `guest_session_id` cookie, random UUID, ~90-day expiry. No `auth.users` row is created for guests — confirmed Option B, no Supabase anonymous auth.
2. **Guest tool-usage ledger** — one shared, generic, Postgres-backed table, parameterized by `tool_key`, reusable by any future guest-accessible tool (§6).
3. **Guest-scoped AI entry point** — a new server action reusing `extractTextFromFile()` and `scoreResumeATS()` unmodified. **Writes nothing resume-shaped anywhere** — no file, no raw text, no parsed data, no score is persisted for a guest, under any circumstance, in Version 1.1.
4. **Analytics events** — one lightweight, generic, Postgres-backed events table (§10).
5. **AI Tools presentation registry** — a small, marketing-layer-only data structure listing which AI tools exist and are linkable vs. "coming soon," consumed by Phase 1B's homepage/nav work (§6, §9).

## 3. Database Changes (Phase 1A)

One new migration, `0015_guest_platform.sql`, two new tables, both fully isolated from the existing schema — **zero changes to any existing table, RLS policy, or Storage bucket.**

```sql
-- Generic guest-usage ledger, shared across ALL current and future guest-accessible
-- AI tools (not just the ATS checker) — see §6 for why tool_key exists now even
-- though only 'ats_checker' is used in Phase 1A.
create table public.guest_tool_usage (
  id uuid primary key default uuid_generate_v4(),
  tool_key text not null default 'ats_checker',
  guest_session_id uuid not null,
  ip_hash text not null,        -- sha256(ip) — raw IP is never stored
  created_at timestamptz not null default now()
);

create index guest_tool_usage_session_idx on public.guest_tool_usage (tool_key, guest_session_id);
create index guest_tool_usage_ip_created_idx on public.guest_tool_usage (tool_key, ip_hash, created_at);

alter table public.guest_tool_usage enable row level security;
-- No policies: written/read exclusively via the service-role client, server-side
-- only. Deny-by-default for anon/authenticated, matching this codebase's existing
-- convention for system-only tables (e.g. audit_logs).

-- Lightweight, generic analytics events — guest and authenticated flows both,
-- so this doesn't need to be rebuilt when future tools ship.
create table public.analytics_events (
  id uuid primary key default uuid_generate_v4(),
  event_name text not null,             -- e.g. 'ats_checker_page_view', 'ats_checker_upload_attempt',
                                         -- 'ats_checker_analysis_success', 'ats_checker_cta_click',
                                         -- 'registration_from_guest_flow'
  guest_session_id uuid,                -- present for guest-originated events
  user_id uuid references public.profiles(id) on delete set null,
  metadata jsonb,
  created_at timestamptz not null default now()
);

create index analytics_events_name_created_idx on public.analytics_events (event_name, created_at);
create index analytics_events_guest_idx on public.analytics_events (guest_session_id);

alter table public.analytics_events enable row level security;
-- No policies: service-role only, same reasoning as above.
```

**Confirmed, final**: `guest_tool_usage` stores only a session id, a hashed IP, a tool key, and a timestamp — enough to enforce "one free scan," nothing else. No resume content, no parsed data, no AI output, no ATS score is stored anywhere for a guest, ever, in Version 1.1. Guest-analysis recovery after registration is explicitly **not** a Version 1.1 feature — if it's wanted later, it gets its own future-version design (likely requiring a deliberate, short-TTL, explicitly-consented cache, which is a different privacy posture than this document's "nothing persists" baseline and shouldn't be backed into now).

## 4. Route Changes (Phase 1A)

- New public page: **`/ai-tools/ats-checker`**, under a new `(marketing)/ai-tools/` route group. Chosen so a second guest-accessible tool later lands at `/ai-tools/<tool-name>` next to this one, with zero URL-structure rework.

- New Server Action file: **`src/lib/analytics/track.ts`** — `trackEvent(name, metadata?)`, fire-and-forget, never blocks or fails the calling request.

## 5. Middleware Changes (Phase 1A)

**None.** Everything outside `/candidate`, `/recruiter`, `/admin` already falls through the existing middleware untouched. The guest session cookie is read/set directly via `cookies()` inside the new server action, not in middleware — zero changes to the one file every route's auth/session-refresh logic depends on.

## 6. Extensibility Design

Ten future tools were named as the long-term vision. Building shared runtime infrastructure for all ten now would be guessing at requirements none of them have yet defined. Two of them already have working backend logic today, just gated behind auth (`resume-improver.ts`, `job-matcher.ts`) — proof the existing per-function pattern in `src/lib/ai/` already scales as tools get added, without needing a shared wrapper.

**What's being built now, because it's cheap and genuinely prevents rework later:**

1. **`guest_tool_usage.tool_key`** (§3) — shared abuse-tracking table, parameterized. Tool #2 reuses it with a new `tool_key` value, zero new migration.
2. **The guest session cookie** — one identity, reusable by any future guest-accessible tool.
3. **`analytics_events.event_name`** — open string, not an ATS-specific enum.
4. **The AI Tools presentation registry** (`src/lib/marketing/ai-tools.ts`) — a typed list:
   ```ts
   interface AIToolListing {
     key: string;
     label: string;
     description: string;
     icon: IconName;
     href: string | null;   // null when status is "coming-soon" — no route exists to link to
     status: "live" | "coming-soon";
   }
   ```
   This is the single source of truth Phase 1B's homepage/nav consumes. It is what makes the "no dead links" rule (§7 below) structurally enforceable rather than a convention someone has to remember.

**What's explicitly not being built now**: a shared `AIToolRunner` abstraction, a plugin system, or stub pages/routes for the other nine tools. When tool #2 is actually specified, it gets its own feature-planning pass under this same workflow — at that point, with two real examples instead of one, it'll be obvious whether a shared runner is warranted. Building it speculatively now is the over-engineering failure mode, not the extensibility win.

## 7. "Coming Soon" Tools — No Dead Links (confirmed policy)

Future tools may be presented on the homepage/nav as "Coming Soon" or "Available Soon," but **a visitor must never be able to click through to a broken or nonexistent page.** Enforced two ways, both driven by the registry in §6:

- **`href: null`** in the registry means the card/nav item renders with no `<Link>` at all — it's a `<button>` or a non-navigating element, visually marked as disabled/upcoming (muted styling, a "Coming Soon" badge), not a clickable route.
- Clicking a "coming soon" item opens a **professional modal** (reusing the existing Radix `Dialog` primitive already in this codebase — see `apply-dialog.tsx` for precedent) explaining the tool is on the roadmap, rather than doing nothing or erroring. This is the better default for anything the registry marks `coming-soon`: acknowledges the click, sets expectations, keeps the experience feeling intentional rather than broken.
- Nothing in the registry is allowed to have both `status: "coming-soon"` and a non-null `href` — that combination is exactly the dead-link case being prevented, and Phase 1B's implementation will treat it as a type-level impossibility (the `href: string | null` shape in §6 already encodes this), not just a convention to remember.

This section governs Phase 1B (first homepage placement of any "coming soon" entries) and continues to apply through Phase 1C.

## 8. Guest Flow (Phase 1A, end-to-end)

1. Visitor reaches `/ai-tools/ats-checker`. `trackEvent("ats_checker_page_view")`.
2. Uploads a PDF/DOCX — same type/size validation values as the authenticated uploader, in a separate component (guest behavior differs materially: no navigation, no persistence, no auth check). `trackEvent("ats_checker_upload_attempt")`.
3. Server action: in-memory burst check → `guest_tool_usage` session-gate check → `guest_tool_usage` IP-cap check (rolling 24h, proposing 3, generous for shared/office NATs) → clear "you've used your free scan" state if blocked, not a generic error.
4. If allowed: `extractTextFromFile()` → `scoreResumeATS()` (reused unmodified) → insert **one** `guest_tool_usage` row (only on a successful score — a transient OpenAI failure must not burn the guest's one free scan) → return the **complete, unhidden** result. `trackEvent("ats_checker_analysis_success")`.
5. Full results render inline — overall score, all sub-scores, weaknesses, suggestions, keyword density, same depth an authenticated user sees — via a new guest-results component (reuses `ScoreRing`; does not repurpose `AtsScoreCard`, which is typed for a persisted DB row a guest result doesn't have).
6. Professional CTA below the results: "Create a free account to save this analysis and unlock unlimited AI tools." `trackEvent("ats_checker_cta_click")` fires on click, before navigating to `/register`.
7. If the visitor registers with a `guest_session_id` cookie present, `trackEvent("registration_from_guest_flow", { guest_session_id })` fires from the existing `registerCandidate` action — one additive analytics call, no change to its validation, DB writes, or return shape.
8. **Confirmed: nothing is carried over automatically.** A guest who registers starts their real resume upload fresh through the existing, unmodified authenticated flow. No re-run, no cache, no import — this was flagged as an open question and is now resolved as out of scope for Version 1.1 entirely.

## 9. Security Considerations (Phase 1A)

- File validation identical to the existing authenticated uploader (PDF/DOC/DOCX, 10MB cap) — same limit values, not duplicated with different numbers.
- Nothing is written to Supabase Storage for guests — no storage-policy surface to secure for this feature.
- IP stored only as SHA-256 hash, never raw.
- Both new tables: RLS enabled, zero policies, service-role-only access — matches this codebase's existing convention.
- Reuses the AI functions' existing graceful-fallback behavior — an OpenAI outage degrades to a placeholder result instead of crashing a guest's one shot.
- Server Actions get Next.js's built-in Origin-check CSRF protection automatically.
- No new environment variables, no new secrets, no change to `SUPABASE_SERVICE_ROLE_KEY` exposure surface.
- Analytics events contain no PII beyond a random session id and, once authenticated, a `user_id` — no email, name, or resume content ever lands in `analytics_events.metadata`.

## 10. Analytics Strategy (Phase 1A)

Lightweight, no new vendor/infra dependency — reuses the existing Postgres database, consistent with how the rest of this codebase is built. Events: page view, upload attempt, successful analysis, CTA click, registration attributed to the guest flow (exact call sites in §8). Conversion-funnel analysis is a simple grouped query over `analytics_events` by `event_name` and day, joined on `guest_session_id` — no dashboard UI is being built in this phase; that's a reasonable fast-follow once there's real data.

## 13. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Guest-scan abuse costs real OpenAI spend | Medium | Postgres-backed session gate (primary) + IP-hash cap (secondary) |
| A failed OpenAI call burns a guest's one free scan | Low-Medium | `guest_tool_usage` row only written on a *successful* score |
| Analytics event writes add latency/failure risk to the guest flow | Low | `trackEvent()` is fire-and-forget, never awaited in a way that can fail the request |
| "Coming soon" entries become dead links if the registry contract is bypassed | Low | Enforced at the type level (`href: string \| null` tied to `status`), not just convention (§6, §7) |
| Phase 1C scope (full brand system) is large and could tempt batching with 1A/1B | Medium | Explicitly sequenced and gated independently — no phase starts until the previous is live-verified |
| Prior untracked SDD draft causing confusion later | Resolved | Archived, not merged, explicitly excluded from this roadmap unless separately re-approved |
