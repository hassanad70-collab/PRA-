## 2. Technical Architecture

### 2.1 Current baseline (unchanged from v1 — re-confirmed, not re-derived)

- Next.js 15.4.11, App Router, React 19, TypeScript strict.
- `src/middleware.ts` → `src/lib/supabase/middleware.ts`: single middleware doing session refresh + role-based route protection.
- Tailwind + CSS custom properties (HSL), dark mode via `.dark` class. Minimal RTL-breaking utility usage found in marketing components (1 occurrence).
- Radix `Dialog` already in use (`apply-dialog.tsx`) — direct precedent for the auth-prompt dialog.
- Guest-friendly gating precedent already exists (`/jobs/[id]`'s "Sign in to apply").
- SEO baseline: `src/app/sitemap.ts`, `src/app/robots.ts` — single-locale today, and will **remain effectively single-URL** under the new i18n decision (see §9).
- AI infra (`src/lib/ai/*`): six hardened functions, foundation for Phase 5.
- Rate limiting (`src/lib/rate-limit.ts`): in-memory, documented serverless limitation — not sufficient alone for guest-scan abuse prevention (§11).
- Migrations `0001`–`0014` applied; next is `0015`.

### 2.2 New architecture components (revised)

1. **i18n layer — `next-intl`, non-routing mode.** next-intl explicitly supports operating without its routing/middleware locale-prefix mechanism: locale is resolved per-request from a cookie via `getRequestConfig` (`src/i18n/request.ts`), not from the URL. This is a first-class, documented next-intl configuration — not a workaround.
2. **No route restructure.** `src/app/` keeps its exact current shape. Zero pages move.
3. **Locale resolution, folded into the existing single middleware**: on any request, if no `NEXT_LOCALE` cookie is present, inspect `Accept-Language` and set one (default `en`). This is a small additive step inside the existing `updateSession` flow — it does not rewrite or redirect the URL, so it composes trivially with the existing auth logic instead of requiring a second middleware layer.
4. **Language switcher**: client component + a small server action (`setLocale(locale)`) that writes the `NEXT_LOCALE` cookie and triggers `router.refresh()` — re-renders Server Components in the new language without any navigation. `localStorage` is used as a client-side cache to avoid a flash-of-wrong-language on subsequent client-side navigations before the cookie round-trips; the cookie remains the source of truth for server rendering (required for correct SSR — `localStorage` isn't readable server-side).
5. **Guest identity layer**: unchanged from v1 — signed, HttpOnly `guest_id` cookie, independent of the locale cookie and of Supabase auth.
6. **`AuthPromptDialog`**: unchanged from v1 — built on the existing `Dialog` primitive.
7. **Guest-scoped AI entry point**: unchanged from v1 — reuses `extractTextFromFile` + `scoreResumeATS`.
8. **`AIToolRunner`**: unchanged from v1 — design/scaffold only.

## 6. Authentication Flow (unchanged from v1)

Middleware + RLS remain the sole authorization boundary. The `AuthPromptDialog` changes *presentation* only:
- Client-side navigation to a protected action → dialog, in place, no navigation.
- Hard navigation to a protected route without a session → middleware redirect to `/login?redirect=...`, exactly as today (a dialog cannot intercept a request before the page exists).
- Recruiter/Admin dashboards: dialog treatment for in-app navigation attempts; hard navigation still redirects server-side — no anonymous "preview" state exists for a dashboard by nature.

## 7. Guest Flow (unchanged from v1, restated for completeness)

1. First request → middleware issues a signed, HttpOnly `guest_id` cookie if absent (separate cookie from `NEXT_LOCALE`; separate concern).
2. Guest browses freely — home (now the premium SaaS homepage), jobs, companies, and (once Phase 4 ships) pricing/blog/about, all at their existing or new top-level URLs, no auth checks in the render path.
3. Guest reaches a protected action → §6 dialog flow.
4. Guest uses the free ATS checker (Phase 3) → §11 abuse-prevention flow, 1 free scan, then dialog prompt.
5. Guest registers → `guest_id` cookie retained; optional backfill of any matching `guest_scans` row into the new `candidates` row (nice-to-have, not required for V1.1).

## 9. SEO Strategy (revised for single-URL i18n)

This is the section most materially affected by removing locale routing — worth reading closely.

- **Metadata**: per-page `generateMetadata()` still uses next-intl's `getTranslations`, so titles/descriptions render in whichever language the *current request's* cookie resolves to.
- **Open Graph / Twitter Cards**: unchanged approach — root `metadata` extended, per-page overrides for jobs/blog/companies.
- **JSON-LD**: unchanged — `Organization`, `JobPosting` (high-value for Google Jobs eligibility), `BreadcrumbList`, `Article` for blog.
- **Sitemap / robots**: unchanged mechanism, simpler output — one URL per page (no `alternates.languages` variants, since there's only one URL per page now).
- **Honest trade-off, stated plainly**: with a single URL serving both languages via cookie-based negotiation, **search engines will predominantly index whichever language is served to an unauthenticated crawler request** — i.e., the default locale (English), since Googlebot won't carry a `NEXT_LOCALE` cookie or meaningfully vary by `Accept-Language` for indexing purposes. This means:
  - **Arabic content will not be separately discoverable via Arabic-language search queries** in this model — there is no distinct, crawlable Arabic URL for Google to index under `hreflang="ar"`.
  - This is a direct, known consequence of the no-locale-URL decision, not an oversight. You've explicitly weighed this against routing/SEO *risk* to the existing production site and chosen to accept it for V1.1, with locale URLs as a deliberate future-version option if Arabic organic search traffic becomes a priority.
  - No mitigation is proposed within V1.1's scope beyond noting it here — the correct fix (locale-prefixed or locale-subdomain URLs with `hreflang`) is exactly the future major version you've already flagged as the intended path.
- **Canonical URLs**: straightforward now — one canonical URL per page, no cross-locale `alternates.languages` complexity to get right.

## 10. Performance Strategy (unchanged from v1)

Server Component-first architecture retained. New homepage sections (Jobs preview, Companies preview) fetch via existing query functions (`getPublishedJobs()` etc.) — no new data-fetching infra needed, just new presentational sections. `loading.tsx` streaming for the AI resume checker's OpenAI call. Bundle discipline: next-intl in non-routing mode still only ships the active locale's message JSON per request (resolved server-side), so the bundle-size profile is the same as the routing approach — this decision doesn't change that benefit.

## 11. Security Review (unchanged from v1)

RLS untouched. Guest-scan abuse prevention: signed `guest_id` cookie (primary) + IP (secondary) + DB-backed check via `guest_scans` (not the in-memory rate limiter, for the same reasoning as v1 — serverless coordination gap matters more when each hit costs real OpenAI spend). No existing control weakened. The `NEXT_LOCALE` cookie is not security-sensitive (no signing needed — worst case of tampering is seeing the wrong language, not an authorization bypass), unlike `guest_id` which must be signed.

## 12. Migration Plan (fully revised — substantially de-risked)

Removing locale-prefixed routing eliminates the single largest risk item from v1's plan entirely. There is no URL migration, no redirect strategy, no sitemap resubmission, no re-pointing of existing backlinks/bookmarks, and no need to touch every existing e2e spec's hardcoded paths.

1. **Migration `0015`** (`guest_scans`) applied via the same `supabase db push` workflow as `0012`–`0014`, verified with the same post-migration schema check.
2. **i18n rollout order**: add the provider/locale-resolution plumbing first (invisible, no UI change), then translate string-by-string, page-by-page — each page's translation is independently verifiable and low-risk since nothing about its URL, route, or data-fetching changes.
3. **Homepage rebuild** (Phase 2) is additive-and-replace within one file (`src/app/page.tsx`) plus new section components — no route change, fully covered by re-running the existing `public-access.spec.ts` suite plus new specs for the added sections.
4. Each phase still ends with the full gate (§14/Phase 8): `npm run lint`, `npm run build`, full Playwright suite against a production build, live Vercel verification — no phase starts until the previous is green.
5. Existing V1.0 e2e specs need **no path changes** under this revised plan (a direct, material simplification versus v1's plan, which required updating every hardcoded path).

## 13. Risks (revised)

| Risk | Severity | Mitigation |
|---|---|---|
| ~~Route restructure breaks bookmarks/SEO/backlinks~~ | **Eliminated** | No longer applicable — no route restructure |
| Arabic content not separately indexable by search engines (single-URL i18n) | Medium, **accepted by explicit decision** | Documented in §9; correct fix is locale URLs in a future major version, already your stated intent |
| RTL layout breakage in components using non-logical Tailwind utilities | Medium | Low existing exposure (1 occurrence found); systematic `ms-/me-`, `start-/end-` audit during Phase 1 |
| Flash of wrong language on first paint before cookie is set (new visitor, no cookie yet) | Low | Middleware sets the cookie on the very first request based on `Accept-Language`, resolved before the page renders server-side — no client-side flash for first-time visitors; returning visitors always have the cookie |

| Blog without a CMS becomes a bottleneck if content velocity is high | Low | MDX now, explicitly revisitable |
